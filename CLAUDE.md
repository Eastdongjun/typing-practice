# 打字练习 — 项目工作规则

## 1. 项目定位

本仓库是打字练习平台唯一受版本控制的需求、设计与源代码库。所有正式需求、决策、数据定义、验收标准和变更记录必须进入本仓库。

## 2. 工作目标

Claude Code 的职责：
- 把业务输入转化为可评审、可开发、可测试、可追踪的需求成果
- 通过 Git 分支和 Pull Request 发布到 GitHub
- 直接实现代码（Electron 桌面应用）

## 3. 信息优先级

1. 已批准的变更记录和决策记录
2. 当前版本 PRD、数据字典、验收标准
3. `PROJECT_CONTEXT.md`
4. 原始需求输入
5. Claude 的合理推断

发生冲突时不得静默覆盖。必须记录冲突、影响和建议处理方式。

## 4. 禁止臆造

缺失信息不得伪装为已确认事实。使用以下标记：

- `已确认`：来源明确且已批准
- `待确认`：必须由业务或技术负责人确认
- `假设`：为了继续工作而采用的临时条件
- `不在范围`：当前版本明确不实现
- `后续版本`：已识别但暂不纳入

所有假设和待确认项写入 `docs/requirements/09-decisions/`。

## 5. 每次需求处理必须输出

根据需求规模至少生成或更新：
1. 需求受理单
2. 背景、目标、范围和边界
3. 用户、角色、职责和数据权限
4. 功能清单及优先级
5. 业务规则、状态机和异常场景
6. 页面/字段/按钮/校验说明
7. 验收标准和测试场景
8. 风险、依赖、假设和待确认问题

## 6. 质量标准

- 每条需求有唯一编号
- 禁止"等等""相关功能""支持一下""优化体验"等不可验收表述
- 规则必须给出边界值、默认值、优先级和冲突处理
- 权限至少覆盖菜单、按钮、字段、数据范围
- 文档更新后执行 `python3 scripts/check_requirements.py`

## 7. 代码质量标准

- HTML/CSS/JS 修改后必须在浏览器验证
- Electron 安全加固：sandbox、contextIsolation、CSP
- 构建前执行 `npm run check`
- 所有文件使用 UTF-8 编码

## 8. Git 工作规则

- 禁止直接向 `main` 推送
- 每次需求使用独立分支：`req/YYYYMMDD-简短名称`
- 功能开发使用：`feat/简短名称`
- 修复使用：`fix/简短名称`
- 提交信息格式：
  - 需求：`docs(requirements): 简要说明`
  - 功能：`feat: 简要说明`
  - 修复：`fix: 简要说明`
- 发布前先执行 `npm run check`
- 默认创建 Pull Request，不自动合并
- 禁止 force push、删除远程仓库

## 9. 安全边界

不得读取、显示或提交：
- `.env`、`.env.*`
- 私钥、证书私钥、访问令牌
- 生产数据库密码
- 客户个人敏感信息

## 10. 完成定义

只有同时满足以下条件才可称为完成：
1. 需求成果齐全（如涉及需求变更）
2. 代码通过手动验证
3. 检查脚本通过（`npm run check` 或 `python3 scripts/check_requirements.py`）
4. 变更已提交到独立分支
5. 分支已推送到 GitHub
6. Pull Request 已创建
7. 已返回 PR 地址、变更摘要和待确认问题

## 11. 项目技术约束

- 前端：纯 HTML/CSS/JS，不引入框架
- 桌面壳：Electron 35
- 构建：electron-builder（NSIS for Windows, DMG for macOS）
- 支持平台：Windows x64 (Win10/11)、macOS Apple Silicon + Intel
- 不支持：移动端、平板、Windows 32位
